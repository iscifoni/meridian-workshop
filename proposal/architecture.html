Frontend State — Composables

useFilters()
Singleton — shared reactive state for all global filters. All views and FilterBar read the same refs.
  • selectedPeriod (ref)
  • selectedLocation (ref)
  • selectedCategory (ref)
  • selectedStatus (ref)
  • hasActiveFilters (computed)
  • getCurrentFilters() → object
  • resetFilters()
useI18n()
Singleton — locale state and translation function. Language persists in localStorage.
  • t(key, params?) → string
  • currentLocale (computed)
  • setLocale(locale)
  • translateProductName(name)
  • translateCustomerName(name)
  • translateWarehouse(name)
useAuth()
Mock user session — hardcoded employee profile and initial task list. No real auth.
  • currentUser (ref)
  •   .name, .email, .role
  •   .department, .location
  •   .tasks (mutable array)
  • isAuthenticated (computed)

Backend Utilities

apply_filters(items, warehouse, category, status)

  • Used by all list endpoints to filter in-memory data. Category and status are case-insensitive; warehouse is exact match. Passing None or 'all' skips that filter.

filter_by_month(items, month)

  • Supports direct month strings (2025-01) and quarter codes (Q1-2025 → Jan–Mar). Applied after apply_filters() in order endpoints.

Key Design Decisions & Constraints

🔒
No authentication
The current user is hardcoded in useAuth.js. There is no login flow, no session management, and no server-side identity. CORS is open (allow_origins=["*"]). This is appropriate for a demo environment only.
💾
In-memory data store — no persistence
All JSON files are loaded at startup. Writes (tasks, purchase orders) survive only for the lifetime of the server process. A restart resets to the file contents. A production path would replace this with PostgreSQL or similar.
🌐
Singleton composables for shared state
useFilters() and useI18n() use module-level refs (outside the function body), making them singletons. Any component that calls these composables shares the same reactive state — this is intentional and how the FilterBar propagates changes to all views without a Vuex/Pinia store.
📐
Options API → Composition API migration (R1)
Reports.vue was the only view still using Options API. It has been migrated to Composition API for consistency. All views now follow the same pattern: setup(), refs, computed, watch, onMounted.
🏗️
No build-time aliasing
Vite is configured without a @/ path alias. All imports use relative paths (../api, ../composables/useFilters). This is intentional — the vite.config.js does not define aliases.
